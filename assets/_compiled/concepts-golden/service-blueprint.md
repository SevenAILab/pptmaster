# 服务蓝图 · Service Blueprint

## 定义

服务蓝图是把用户行为、前台触点、后台支持、系统流程和证据物串起来的服务设计工具,用于看清一段体验如何被组织交付。

本概念在 Phase 1 中主要来自全景图 06-体验层的 "服务蓝图 Service Blueprint",属于 MAP-STRUCTURE 新增候选,缺少 247 书中的完整专章。结合品牌全案框架和用户旅程资料,它在 PPTAgent 中用于把用户旅程中发现的痛点拆到可执行流程: 用户看到什么、员工做什么、后台系统如何支持、哪些环节会掉链子。

一句话: 用户旅程看用户经历,服务蓝图看品牌如何交付这段经历。

## 适用场景

- **用户体验诊断**: Sub-Agent ① 在旅程地图后定位服务卡点
- **品牌体验落地**: Sub-Agent ⑤ 将品牌承诺转成前台服务与后台流程
- **会员/售后/私域流程设计**: 设计咨询、购买、履约、售后、复购的完整链路
- **年度体验优化**: Sub-Agent ⑥ 将体验问题转为流程项目
- **不适合**: 只做传播策略时强行使用；服务蓝图适合真实触点和流程优化

## 使用步骤

### 第 1 步 · 选定服务场景

明确要画哪一段流程,例如首次购买、定制咨询、售后退换、会员复购或线下体验。

输出标准: 场景边界 + 起点/终点 + 目标用户。

### 第 2 步 · 写用户行为

沿时间线列出用户每一步做什么、想什么、担心什么。优先承接 User-Journey。

输出标准: 用户行为时间线。

### 第 3 步 · 标注前台触点

列出用户能看到和接触到的内容: 页面、客服、包装、短信、社群、门店人员、说明书等。

输出标准: 前台触点清单。

### 第 4 步 · 拆后台支持

列出用户看不见但决定体验的流程: 库存、订单、客服 SOP、物流、质检、CRM、权限和供应链。

输出标准: 后台流程和责任人。

### 第 5 步 · 找失败点和证据物

标出等待、误解、重复填写、无人负责、信息不一致等失败点,同时列出需要的证据物。

输出标准: 痛点/断点/证据物表。

### 第 6 步 · 形成优化方案

把断点转成流程、话术、系统、内容和指标优化动作。

输出标准: 优化动作 + 负责人 + 指标。

## 输入输出示例

### 示例 1 · 植愈坊敏感肌购买服务蓝图 (247 书贯穿教学案例)

> **案例可追溯报告**: 植愈坊是 Seven 247 书全书贯穿的**教学性虚拟案例** (非真实商业体)。具体出处行号待 Codex 后续按 P0-4 模板填充, 当前仅标注教学属性。**禁止包装成"真实品牌成功案例"**。

背景: 用户在小红书被种草后,担心敏感肌是否能用,需要完成咨询、购买、试用和复购。

| 层级 | 内容 |
|---|---|
| 用户行为 | 看笔记 → 咨询敏感肌 → 选择香型 → 下单 → 试用 → 反馈 |
| 前台触点 | 小红书笔记、客服回复、香型选择页、试用指南、售后私信 |
| 后台支持 | 成分知识库、客服 SOP、订单备注、试用反馈表、CRM 标签 |
| 失败点 | 香型选择困难、敏感肌风险不确定、售后反馈没人跟进 |
| 优化 | 增加敏感肌 FAQ、无香基础款推荐、24h 试用提醒 |

**关键洞察**: 植愈坊的体验优势不只在产品,也在咨询和试用流程能不能让用户安心。

### 示例 2 · 假设在线问诊 SaaS X 的服务蓝图 (Tier 3 抽象示例)

> **案例可追溯报告**: 抽象演示案例 (非真实品牌, 用于演示服务蓝图前后台协同结构). 任何与真实品牌的相似性纯属巧合。本示例不应被引用为"真实案例佐证"。

背景: 假设一家在线问诊 SaaS X 想优化"用户从挂号到拿到诊断结果"的全流程:

| 层级 | 内容 |
|---|---|
| 用户行为 | 描述症状 → 选医生 → 排队 → 视频问诊 → 拿处方 → 复购药品 |
| 前台触点 | 症状自检表单、医生筛选页、排队页、视频通话、处方下载、复购入口 |
| 后台支持 | 医生排班系统、问诊知识库、视频通话基础设施、电子处方系统、药品配送对接 |
| 失败点 | 症状描述太复杂、医生选错、排队过久、视频卡顿、处方看不懂 |
| 优化 | AI 辅助症状归类、医生匹配算法、预约制减少排队、视频备份、处方注释 |

**关键洞察**: 服务蓝图的核心价值是**让前台体验问题暴露后台支撑断点** — 用户的卡点 (e.g. "视频卡顿") 90% 是后台问题 (e.g. 视频基础设施容量不足), 单纯优化前台 UI 解决不了。

## 常见误用

1. **把服务蓝图画成用户旅程**: 只写用户行为,没有前后台流程 → 看不到组织交付问题
2. **场景过大**: 从认知到忠诚全画一张 → 细节无法落地
3. **没有后台责任**: 只列触点,不列系统和负责人 → 优化无法执行
4. **忽略证据物**: 用户看不到承诺被兑现的证据 → 信任感不足
5. **只画现状不改流程**: 蓝图漂亮但没有优化动作 → 变成咨询装饰
6. **不绑定指标**: 不知道优化后看等待时长、转化率还是满意度 → 无法复盘
7. **没有更新机制**: 服务流程变化后蓝图不更新 → 团队执行偏离
