# JTBD · Jobs to be Done · 待办任务理论

## 定义

JTBD (**Jobs to be Done**) 是由哈佛 Clayton Christensen 推广的用户需求分析理论:

> **"用户'雇佣' (hire) 一个产品来完成某个'任务 (Job)'"**

核心洞察:
- 用户买的不是产品本身, 而是**雇佣产品帮自己完成某项任务**
- 痛点 (Pain) = 用户在完成这个任务过程中遇到的困难
- 收益 (Gain) = 这个任务完成后用户获得的价值/感受

**第一性原理隐喻** (摘要 06, 福特经典):
> "如果我问消费者想要什么, 他们会说想要一匹更快的马。
> '更快的马'是用户能说出口的需求, 基于他们已有认知的想象。
> 但消费者真正要的不是'马', 是'更高效的出行'。
> 因为前者是路径, 后者才是目标。**从马到车, 中间隔着的那一步, 才是洞察。**"

JTBD = 从"用户说的需求 (路径)"挖到"用户真正要的目标 (Job)"。

## 适用场景

- **消费者洞察案的核心骨架**: Sub-Agent ① consumer_insight 必检字段"≥3 个核心 JTBD"的产出依据
- **真需求 vs 伪需求判断**: 摘要 05 强调"画像是入场券, 需求才决定上限"
- **价值主张设计的需求侧输入**: BMC 的 VP 模块 / Value-Prop-Canvas 的左侧"客户简档"
- **品牌定位的差异化机会发现**: 找现有产品没解决好的 Job → 差异化空间
- **新产品/新功能立项判断**: 是否对应一个真实 Job? Job 是否够大够普遍?
- **不适合**: 替代用户画像 / 用户旅程 (这些是 JTBD 的下游展开工具)

## 使用步骤

### 第 1 步 · 找准目标用户 (摘要 05 核心)

战略就是**聚焦和取舍**。你选择服务谁, 就决定了你不服务谁, 这定义了**生意边界和战略重心**。

**铁律**: 总是想要讨好所有人的话, 就会导致谁都服务不好。

**案例对照**: 拼多多早期聚焦"价格敏感型用户", 不做消费升级, 不拼物流 — 所有资源聚焦把价格打到最低 → 用户定义清楚了, 品牌/产品/渠道/供应链/运营才能形成合力。

输出: 1-3 个具体目标用户群 (粒度细到 Persona-5W2H 程度)

### 第 2 步 · 识别用户的 Job (任务)

对每个目标用户群, 列出他们"想完成的任务":

**Job 的三层结构** (摘自 Christensen 经典理论):
- **功能性 Job (Functional Job)**: 用户要完成的实际任务 (e.g. "清洁我的皮肤")
- **情感性 Job (Emotional Job)**: 用户在完成功能时希望获得的感受 (e.g. "感到被关爱、被疗愈")
- **社交性 Job (Social Job)**: 用户希望别人怎么看自己 (e.g. "被认为是讲究品质的人")

**Job 提炼语法**:
```
当 [情景/触发] 时,
我 [想要的目标],
以便 [更深的目的/价值]。
```

示例:
- 当我下班疲惫回家时, 我想用一块手工皂洗澡, 以便快速从工作压力中切换到自我修复状态
- (不要写: "我想买好用的手工皂" — 这是路径, 不是 Job)

### 第 3 步 · 真需求 vs 伪需求三大判定维度 (摘要 05 独家)

很多人把用户细分做得很细, 标签做得漂亮, **但如果本身是伪需求, 千斤拨四两**。

| 维度 | 真需求 | 伪需求 |
|---|---|---|
| **强度** | 痛点够痛, 用户愿意持续付费 | 痛点不痛, 用户犹豫掏钱 |
| **普遍性** | 大量用户都遇到 | 小众或仅特定场景 |
| **持久性** | 长期持续存在 | 短期热点, 一过即灭 |

**经典对照 (摘要 05)**:
- **真需求 网约车**: 解决了打车难、等车久、价格不透明、路线无法实时追踪。痛点够痛够普遍, 用户愿意持续付费 ✅
- **伪需求 共享单车头盔**: 安全确实重要, 但用户**不愿戴不知多少人戴过的公共头盔** (嫌脏嫌麻烦), 政府也无强制政策 ❌

### 第 4 步 · 痛点 (Pain) + 收益 (Gain) 双侧挖掘 — 247 书 ch04 + ch08

针对每个核心 Job, 进一步挖:

**痛点 (Pain) - 完成 Job 过程中的困难**:
- 用 **Needs/Wants/Fears 三层分析法**: 痛点往往和用户的"恐惧"相关 (恐惧皮肤受损 / 恐惧浪费钱 / 恐惧选择错误 / 恐惧麻烦)
- 数据来源: 深度访谈 + 用户抱怨/差评 + 场景观察 + AI 分析用户文本

**收益 (Gain) - Job 完成后的价值**:
- 必备收益 (Must-have): 用户期望基本满足
- 期望收益 (Expected): 用户期望显著满足
- 渴望收益 (Desired): 用户没说但很想要
- 出乎意料收益 (Wow): 用户没想到却获得了

### 第 5 步 · 输出 — 核心 JTBD 清单 + 痛点-收益地图

最终交付给 Sub-Agent ① 的输出:

- **3-5 个核心 JTBD** (1 句话格式: "当...时,我想要...,以便..." )
- 每个 JTBD 配套:
  - 2-3 个核心痛点 (优先列高强度高普遍的)
  - 2-3 个核心收益 (优先列期望/渴望级)
  - 当前用户的应对方式 (替代方案/绕行方案)

Sub-Agent ① 渲染时强烈推荐版式 **S04 Six Cells** (6 个 JTBD 一图) 或 **S15 Matrix** (痛点 × 收益矩阵)。

## 输入输出示例

### 示例 1 · Dropbox 创立故事 (247 书 ch04 经典)

> **案例可追溯报告**: 真实公知品牌。本示例核心叙述 ("Drew Houston 出门忘带 U 盘 / 创造 Dropbox 解决跨设备文件同步痛点") **完全来自** `assets/_raw/books/0to1-brand/ch04-user.md` L596-602 原文 (Seven 在 247 书 ch04 给出的 Dropbox 痛点案例)。JTBD 结构化 ("当我...我想...以便...") + 收益分层 (期望/渴望/Wow) 是 Claude 基于该原文做的 JTBD 框架化拆解, 不是 247 原文用词。

**目标用户**: 经常需要在不同设备间同步文件的知识工作者 (创始人 Drew Houston 自己就深受其苦)

**核心 JTBD**:
- 当我出门时, 我想随时随地访问任何设备上的文件, 以便不被"忘带 U 盘"打断工作

**痛点**:
- U 盘容易忘带/丢失
- 邮件附件版本管理混乱
- 跨设备文件同步麻烦

**收益**:
- 无感同步 (期望级)
- 任何设备秒访问 (渴望级)
- 协作时一份文件多人编辑 (Wow 级)

**应对方式 (当时)**: 邮件给自己 / 多设备各拷一份 / 随身带 U 盘 → 都很笨重

**Dropbox 答案**: 极简云同步, 拖入即用, 跨设备秒同步 → 解决核心 Job + 部分 Wow 收益 → 迅速获得大量用户

### 示例 2 · 植愈坊手工皂 JTBD 推演 (Claude 基于 247 书 ch04 推演)

> **案例可追溯报告**: 植愈坊是 Seven 247 书全书贯穿的**教学性虚拟案例**。247 书 ch04 §"痛点理论小科普"L613-616 原文用植愈坊"温和清洁皮肤并带来愉悦感"作为 JTBD 隐喻,但**没有完整展开 3 个核心 JTBD + 痛点 + 收益**。本示例的 3 个 JTBD (功能/情感/社交) + 痛点/收益分层是 Claude 基于 247 书 ch04 植愈坊上下文 + JTBD 标准框架综合推演。**禁止包装成"植愈坊真实用户调研结果"**。

**目标用户**: 都市年轻女性, 25-35 岁, 注重情绪疗愈与生活仪式感

**3 个核心 JTBD**:

```
JTBD 1 (功能):
  当我下班疲惫洗澡时,
  我想用一块手工皂温和清洁皮肤,
  以便不刺激敏感肌+恢复皮肤舒适。

JTBD 2 (情感):
  当我感受到生活压力大时,
  我想通过洗澡的 15 分钟仪式,
  以便从工作切换到自我修复, 获得疗愈感。

JTBD 3 (社交):
  当朋友来家里看到我的洗手间时,
  我想让看到的人觉得这个细节有品味,
  以便强化"我是讲究品质的人"的自我形象。
```

**核心痛点 (JTBD 1)**:
- 市面普通皂含化学添加 → 恐惧伤肤 (Fear)
- 香味太浓或人工香精味 → 不喜欢
- 起泡虚浮 → 体验不达预期

**核心收益 (JTBD 2)**:
- 天然精油的真实香气 (期望)
- 仪式感包装让"打开"也成为体验 (渴望)
- 触感与香气搭配带来短暂"心流" (Wow)

**当前应对**: 用大众沐浴露 (功能解决但情感缺失) / 买高端品牌 (溢价但仪式感不足)

**植愈坊机会**: 同时解决功能 + 情感 + 社交三层 Job, 这是大品牌没做透的位置

### 示例 3 · 反例 共享单车头盔 (摘要 05 经典)

> **案例可追溯报告**: 真实公知反例。本示例核心叙述 ("共享单车头盔是伪需求 / 用户嫌脏嫌麻烦 / 无强制政策") **完全来自** `assets/_raw/methodologies/summaries/05-user-analysis.md` L42 原文 (摘要作者关于"网约车真需求 vs 共享头盔伪需求"的对比)。强度/普遍性/持久性三维度判断框架是 Claude 基于该原文做的 JTBD 检验法归纳, 不是摘要原文的检验框架。

看似有需求 (安全), 但 JTBD 检验失败:
- 强度: 用户更在意"嫌脏嫌麻烦", 戴公共头盔意愿低 → 弱
- 普遍性: 短途骑行场景, 用户不觉得高危 → 普遍性低
- 持久性: 没有强制政策 → 不稳定

→ 失败的 Job 假设 → 失败的产品立项

## 常见误用

1. **JTBD 写成产品描述**: e.g. "用户想要好用的手工皂" → 这是"路径"不是"Job", 应该是"洗澡时获得清洁+疗愈"
2. **只挖功能 Job, 忽视情感/社交 Job**: → 产品做出来"够用但不打动" → 用户没复购理由
3. **JTBD 太宽**: e.g. "用户想要美好生活" → 太大无法落地 → 必须细化到具体场景+具体行为
4. **混淆 JTBD 和用户画像**: 画像告诉你用户"长什么样" (人口学/心理), JTBD 告诉你用户"想完成什么" (任务) — 摘要 05: "画像是入场券, 需求才决定上限"
5. **凭直觉编 JTBD, 不做用户访谈**: 必须来自真实用户口语 + 行为观察 + 差评分析, 否则就是"产品经理自嗨"
6. **跳过真需求 vs 伪需求判定**: 看见"用户提了一个需求"就当真需求 → 像 Touch Bar / 共享单车头盔 → 浪费资源
7. **静态化使用**: 用户的 Job 会随生活阶段/技术演进而变化 (e.g. "更快的马" → 汽车 → 共享出行 → 自动驾驶) → 必须持续观察迭代
8. **忽视"应对方式"**: 不分析用户当前怎么解决这个 Job → 找不到差异化突破口 (Dropbox 找到了 U 盘/邮件附件的应对方式才能突围)
